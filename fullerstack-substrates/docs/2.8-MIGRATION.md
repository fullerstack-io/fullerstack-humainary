## What changed in the upstream API

### NEW — `Ticker` interface (SPEC §11.5)

Circuit-owned periodic emitter. Created from a `Circuit`:

```java
Ticker t = circuit.ticker(Duration.ofSeconds(1), targetPipe);
Ticker t = circuit.ticker(name, Duration.ofSeconds(1), targetPipe);
```

Contract highlights (SPEC §11.5):

- Emits monotonic gap-free `Long` sequence numbers (0, 1, 2, …) into the target pipe.
- Fixed-rate scheduling on a grid — a late tick MUST NOT shift the grid;
  scheduling error MUST NOT accumulate.
- Bounded catch-up — if scheduling falls more than one interval behind,
  re-anchor (no burst of skipped ticks; sequence remains gap-free).
- Circuit-admitted delivery — each tick goes through the ingress queue,
  processed in circuit-context confinement.
- No tick shedding — backlog accumulates; coalesce/throttle with Fiber
  rate-control operators (`every`, `diff`, `reduce`) if needed.
- Failure isolated per §15.4 — receptor throw doesn't stop the ticker.
- Anchored tenure; is a Resource (closeable). Closing the circuit stops
  further submissions.

### NEW — `Flow.scan(initial, step)` 2-arg overload

```java
default <S> Flow<I,S> scan(Supplier<? extends S> initial,
                            BiFunction<? super S, ? super O, ? extends S> step) {
  return scan(initial, step, Function.identity());
}
```

A default method on the `Flow` interface — drops the `Function.<S>identity()`
boilerplate when the projection is identity. We get it for free; the 3-arg
form `FsFlow.scan` is unchanged.

### REFINED — sealed `Lookup` hierarchy

```java
sealed interface Lookup<T> permits Bank, Pool
non-sealed interface Bank<R extends Resource<R>> extends Lookup<R>, ...
non-sealed interface Pool<T> extends Lookup<T>
```

Compile-time refinement only. **Zero impl changes were needed** for
fullerstack-substrates — `FsBank` implements `Bank<Conduit<E>>`,
`FsConduit` implements `Conduit`, `FsDerivedPool` implements `Pool<T>`.
Nothing implements `Lookup` directly. The audit (commit immediately
before the FsTicker work) confirmed this.

### Documentation / SPEC clarifications

- `Flow.scan` state-as-output explicitly permitted (state and output type
  can be identical — confirms our existing `Function.identity()` use).
- New conformance requirement #18 for Ticker scheduling guarantees.
- Capability matrix, identity-bearing tables, open-required-operations
  all augmented with Ticker.
- Provider-mismatch detection added to §15.1 for `Circuit.ticker` target pipe.

## Phase 2 — `Flow.scan(initial, step)` 2-arg overload

Free via Java default method dispatch — `FsFlow` inherits the new overload
from `Flow`, and it routes back to our existing 3-arg `scan` with
`Function.identity()` as the projection. No implementation work.

Downstream: `backtest/` migrated 21 holons from the verbose 3-arg form to
the new 2-arg form (separate commit; not load-bearing for the SPI side).

## Phase 3 — `FsTicker` implementation

New class: `io.fullerstack.substrates.FsTicker` (105 lines).

### Design decisions

| Decision | Choice | Rationale |
|---|---|---|
| Scheduler mechanism | Single `ScheduledExecutorService` (daemon, single-thread) per circuit, shared across all that circuit's tickers | Cheapest mechanism that satisfies fixed-rate scheduling; tick callbacks just enqueue onto the ingress queue, so we don't need parallelism on the scheduler side |
| Grid anchoring | `anchorNanos` set at construction; each tick computes `nextDeadline = anchor + (seq+2) * intervalNanos` | Late ticks compensate by shortening the next wait — no drift accumulation |
| Re-anchor threshold | `delay < -intervalNanos` (more than one interval behind grid) | Matches SPEC §11.5 "bounded catch-up" wording — re-anchor only on real stalls, not minor scheduling jitter |
| Sequence numbering | Per-ticker `long`, incremented on the scheduler thread | Scheduler is single-threaded per circuit; no contention; sequence stays gap-free across re-anchors |
| Failure isolation | `try { target.emit(seq) } catch (Throwable ignored)` | Per SPEC §15.4 + §11.5 — receptor failures isolated; ticker continues |
| Lifecycle | `closed` volatile flag + cancellable `ScheduledFuture`; `closeAwait` uses `circuit.checkExternalCaller` + `circuit.await` | Matches the pattern used by `FsTap`, `FsConduit`, `FsBank` |
| Provider mismatch | `target instanceof FsPipe` check in `Circuit.ticker(...)` factory | Per SPEC §15.1 provider-mismatch on `Circuit.ticker` — fails fast at creation, not at first tick |

### `FsCircuit` changes

- New imports: `java.time.Duration`, `java.util.concurrent.{Executors, ScheduledExecutorService}`, `Substrates.Ticker`.
- New volatile field `scheduler` (lazy single-thread daemon executor).
- New accessor `ScheduledExecutorService scheduler()` (lazy-creates on first use).
- New factory methods `ticker(Duration, Pipe)` and `ticker(Name, Duration, Pipe)`.
- `close()` now also calls `scheduler.shutdownNow()` after the close marker is queued.

## Phase 5 — Performance baseline (spot-check)

PipeOps + CyclicOps re-measured against 2.8 (Codespaces 2-vCPU host,
JDK 26, JMH `-f 1 -wi 3 -i 5 -w 2s -r 2s`, same harness as 2.7 spot-check).

**PipeOps (2.7 → 2.8):**

| Benchmark | 2.7 ns/op | 2.8 ns/op | Δ |
|---|---:|---:|---:|
| `baseline_blackhole` | 0.971 | 0.776 | −20% (noise) |
| `baseline_receptor` | 0.923 | 0.858 | −7% (noise) |
| `baseline_counter` | 3.981 | 3.029 | −24% |
| `pipe_create_chained` | 2.402 | 2.221 | −8% |
| `pipe_create` | 8.932 | 7.889 | −12% |
| `async_emit_single` | 16.253 | 15.727 | −3% |
| `async_emit_batch_await` | 21.063 | 15.787 | −25% |
| `async_emit_chained_await` | 22.520 | 36.826 | +63% (CI ±126 — noise) |
| `async_emit_batch` | 25.350 | 35.916 | +42% (CI ±56 — noise) |
| `async_emit_fanout_await` | 29.565 | 21.231 | −28% |
| `pipe_create_with_flow` | 30.061 | 27.020 | −10% |
| `async_emit_with_flow_await` | 46.707 | 32.022 | −31% |
| `async_emit_single_await` | 194.918 | 114.459 | −41% |

**CyclicOps (2.7 → 2.8):**

| Benchmark | 2.7 ns/op | 2.8 ns/op | Δ |
|---|---:|---:|---:|
| `cyclic_emit` | 1.507 | 1.596 | +6% (noise) |
| `cyclic_emit_batch` | 1.590 | 1.833 | +15% (noise) |
| `cyclic_emit_deep_await_batch` | 17.179 | 14.856 | −14% |
| `cyclic_emit_deep_await` | 18.082 | 14.847 | −18% |
| `cyclic_emit_await` | 27.062 | 19.442 | −28% |
| `cyclic_emit_await_batch` | 32.202 | 22.775 | −29% |

**Reading the table:** every PipeOps row except `async_emit_chained_await` and
`async_emit_batch` improves or holds steady — those two have confidence intervals
larger than their score (±126 and ±56 ns) and were already that noisy at 2.7, so
the apparent regression is iteration-to-iteration variance on shared Codespaces
hardware, not a real change. Every CyclicOps row is within noise or improves.

**Conclusion: no regression detected on the hot path.** Several benchmarks show
modest improvements, most likely from upstream JIT compilation behavior on the
2.8 jar (the SPI hot paths — `FsPipe.emit`, `FsChannel.dispatch`, `IngressQueue`,
`TransitQueueRing` — were not modified for 2.8). Full data:
`benchmark-results/2.8-pipeops-cyclicops.txt`.

No Ticker benchmark yet — would require designing a meaningful scheduled-emission
metric (mean inter-tick latency? grid-drift error? scheduler overhead per tick?).
Tracked as follow-up.
