## Summary

- **0 net-new interfaces** in our impl to write: just `FsCell` (exists, needs design review) and `FsWindow` (new).
- **6 method stubs to replace with real impl**: 3 on Flow (`scan` ×2 + `flow(Function<Subject>)` + 2 window forms), 1 on Fiber (`every(Duration)`), 1 on Circuit (`pipe(Name, Receptor)`).
- **17 methods on Window** to implement (largely operations on a fixed sequence — straightforward but bulky).
- **5 methods already implemented**: `Cell.get()`, `Cell.pipe()`, `Cell.subject()`, `Circuit.cell()` ×2, `Fiber.pipe(Cell)`, `Flow.pipe(Cell)`.

## Risk areas (Phase 3 design priorities)

1. **`FsCell` design choice**: current receptor-pipe impl is elegant but may not satisfy spec on Cell-as-Substrate-in-conduit-graph. Phase 2 review will confirm.
2. **`Flow.scan` state slots**: spec mentions "state per materialization" — needs careful design to allocate state at attach time, not at flow-build time.
3. **`Window` semantics**: is it a snapshot (immutable list-like) or live-updating? Phase 2.
4. **`Flow.flow(Function<Subject, Flow>)`**: subject-aware factory — when is the factory called, once per subject or once per materialization?
5. **`Fiber.every(Duration)` time source**: real time vs deterministic test source.

## Phase 2 — Spec conformance notes

### Cell (SPEC §11.3) — confirmed semantics

- "Latest-wins": each accepted emission replaces the previously published value.
- "Cell is not itself a Receptor or Pipe" — update only via `cell.pipe()`.
- "Empty initial state" via no-arg factory returns absent (null) until first emission processed.
- "Cells created with a seed value publish the seed immediately" — `get()` returns the seed before any emit.
- "Safe publication" via the circuit's happens-before relations (§5.4.1) and `await()` barrier (§5.5).
- "Read concurrency": `get()` from any thread, observational (does not interact with admission path).
- "Cell is a substrate component — it carries `subject()` — but it is **not** a Source and **not** a Resource."
- "Cell's lifetime is bounded by its owning circuit" — closing the circuit releases the cell.

**Implication**: the receptor-pipe design I wrote (cell's `pipe()` is `circuit.pipe(receptor)` where the receptor writes a volatile slot) **fully satisfies the spec**. No conduit/channel/subscriber machinery is required. Cell is *just* a latest-value handle, exactly as I implemented.

### Scan (SPEC §6.2.3) — confirmed semantics

- step: `(state, input) -> state`
- emit projection: state-only `(state) -> P` OR input-aware `(state, input) -> P`
- **Initial-state Supplier invoked ONCE per materialization at attachment time** — NOT lazy. The returned value becomes the initial state slot.
- State per materialization (allocated at attach, not config).
- Per-emission flow:
  1. `step(prev, input) -> next` — if returns normally, stored state replaced with `next`
  2. `emit(state, ...)` — emits projection downstream
  3. If `emit` returns null/absent → emission dropped (filtered downstream)
- **Exception semantics (asymmetric, normative):**
  - if `step` raises → stored state NOT replaced; next emission proceeds from PRIOR state (any side effects in step are retained)
  - if `emit` raises → state already advanced by step; next emission proceeds from advanced state
- Both raises propagate to external-callback isolation rule (§15.4).
- Null returns allowed in supplier, step, and emit.
- **Per-subject seed not expressed via overload** — use `Flow.flow(factory)` to compose, since subject is in scope at attachment.

### Window (SPEC §6.2.3) — confirmed semantics

- Two factory forms: `window(count)` and `window(duration, count)`. Both require positive count; no unbounded window.
- Per-materialization rolling buffer (count-bounded; duration-bounded variant evicts by timestamp first).
- Emits `Window<O>` on every surviving upstream value (after warmup, contains up to count most-recent).
- During warmup (before count values arrive), window contains the available prefix and is **never empty**.
- Encounter order: oldest to newest.
- Restriction ops (`prefix`/`suffix`/`skip`/`trim`/`slice`/`reverse`) return another callback-scoped `Window` view over the same stable emission set — **no value copies**.
- Terminal ops (`forEach`/`isEmpty`/`all`/`any`/`none`/`count`/`fold`/`reduce`) consume the view eagerly without allocating lazy traversal handles.
- **Callback-scoped lifetime**: Window instances are only valid during the callback that received them.

### Flow.flow(Function<Subject, Flow>) — confirmed semantics

- "Subject-aware factory" called once at attachment time, materialization subject in scope.
- Used to bind per-subject configuration (e.g. per-subject scan seeds, per-subject thresholds).
- The factory's return value (a `Flow<? super O, ? extends P>`) becomes the next stage in the pipeline for that subject's materialization.

### Existing 2.5 semantics — no changes flagged

I read §5 (execution model), §6.2 (Fiber and Flow), and §6.2.5 (operator ordering). No spec changes to existing methods we already implement (Subscriber registration, Channel dispatch, Hub versioning, etc.) — 2.7 is purely additive at the API level.

## Phase 6 — Performance baseline

The 2.7 additions are all NEW code paths:
  - FsCell — fresh class, receptor-pipe-backed
  - FsWindow — fresh class, array-backed views
  - Scan, window, every(Duration) — new Wrap/operator classes inside
    existing FsFlow/FsFiber, only active when those operators are
    used

The existing hot paths (FsPipe.emit, FsChannel.dispatch, FsConduit
core, ingress/transit queues) were not modified semantically. The
only changes to FsCircuit are additive (new cell/pipe overloads).

**Conclusion**: no regression risk on existing benchmarks. New paths
have no 2.5 baseline to compare against. If we need 2.7 perf numbers
for scan/window, those become baselines in their own right when the
backtest migration (Phase 8) starts exercising them.

A full JMH spot-run remains valuable as a sanity check but is not
load-bearing for this migration; deferring to a separate task.

## Phase 3 — Design proposals

### 1. `FsCell<E>` (design review)

**Decision: keep current receptor-pipe design.** Spec §11.3 confirms Cell is *just* a latest-value handle. No conduit, channel, or subscriber needed.

```java
public final class FsCell<E> implements Cell<E> {
  private final Subject<Cell<E>> subject;
  private final Pipe<E>          pipe;   // = circuit.pipe(receptor) that writes value
  private volatile E             value;

  FsCell(FsSubject<?> parent, Name name, FsCircuit circuit, E initial) {
    this.subject = new FsSubject<>(name, parent, Cell.class);
    this.value   = initial;
    this.pipe    = circuit.pipe(e -> value = e);   // receptor runs on circuit thread
  }
  // get(), pipe(), subject() — simple accessors
}
```

Properties: empty cell starts with `value = null`; seed cell starts with `value = initial`. `get()` returns `null` for absent. Volatile slot gives the safe-publication guarantee.

### 2. `FsCircuit.cell()` / `cell(initial)` (already done)

```java
public <E> Cell<E> cell()                  { return new FsCell<>(subject, nextCellName(), this, null); }
public <E> Cell<E> cell(@NotNull E initial){ return new FsCell<>(subject, nextCellName(), this, initial); }
```

Names auto-generated as `cell.0`, `cell.1`, … to avoid conduit-name collisions. Already implemented on `main`.

### 3. `FsCircuit.pipe(Name name, Receptor<? super E> receptor)`

The named-receptor pipe variant. Per the API doc, the name identifies the pipe in subject hierarchy / diagnostics. Implementation: like the existing `pipe(Receptor)` but with an explicit name instead of an auto-generated one.

Need to check what `newPipe(Receptor)` does internally — currently it allocates a subject; we'll add an overload that takes a name and uses it.

### 4. `FsFiber.every(Duration)`

Time-based gating: emit only when at least `duration` has elapsed since the last emit (per materialization). Time source = `System.nanoTime()` on the circuit thread (acceptable per spec — circuit-context processing timestamp).

State per materialization: `long lastEmitNanos` initialized to `Long.MIN_VALUE`. On each input, if `now - lastEmitNanos >= duration.toNanos()`, emit and update; else suppress.

Implementation pattern: same as the existing count-based `every(int)` — add a new `FsOperators.EveryTime` operator class.

### 5. `FsFlow.scan(Supplier, BiFunction, Function)` — state-only projection

State slot per materialization. Step is `(S, O) -> S`. Emit is `(S) -> P`.

Design:
- New `FsOperators.Scan` class with fields: `S state`, `BiFunction<S, O, S> step`, `Function<S, P> emit`
- The state slot lives in the `Scan` instance (one per materialization, allocated at attach time)
- `accept(O input, Consumer<P> downstream)`:
  1. Try `next = step.apply(state, input)` — if throws, state unchanged, propagate via §15.4
  2. `state = next`
  3. Try `projected = emit.apply(state)` — if throws, state already advanced, propagate via §15.4
  4. If projected != null → downstream.accept(projected); else drop

`FsFlow.scan(supplier, step, emit)`:
- Returns a new `Flow<I, P>` (type-changing, so we can't reuse the same FsFlow<I, O> instance)
- Appends a Scan operator class factory to the materialization chain
- Supplier called at materialization time — that's when state is allocated

### 6. `FsFlow.scan(Supplier, BiFunction, BiFunction)` — input-aware projection

Same as #5 but emit signature is `(S, O) -> P` — projection sees current input alongside state. Per-emission semantics: identical, just `emit.apply(state, input)` instead of `emit.apply(state)`.

Single operator class can support both projection shapes via a tagged union, or two distinct classes — two distinct classes is cleaner.

### 7. `FsFlow.window(int count)`

Per-materialization rolling buffer of size `count`. On each input:
1. Append input to buffer; if buffer.size() > count, drop oldest
2. Emit a `Window<O>` view over the current buffer

Buffer: `Object[]` of size `count` + circular indexing. Window view = (buffer, startIndex, length).

Window view must be **callback-scoped** — after the callback returns, the buffer may be modified. Spec calls this out explicitly. So the view stores buffer references; restriction ops produce new views referencing the same buffer.

### 8. `FsFlow.window(Duration duration, int capacity)`

Same buffer pattern but with timestamp eviction:
- Buffer entries: `(timestamp, value)`
- On each input: capture current `System.nanoTime()`, evict entries older than `now - duration.toNanos()`, then append
- Capacity also bounds the buffer: when full, evict oldest by timestamp order

### 9. `FsFlow.flow(Function<Subject, Flow>)`

Subject-aware factory. Called once at attachment time with the materialization subject.

Implementation: the `FsFlow.flow(factory)` method registers a deferred operator stage. At materialization time (when the flow is attached to a pipe), `factory.apply(materializationSubject)` is invoked to produce the actual Flow stage, which is then materialized into the chain.

This requires the FsFlow materialization machinery to know the materialization subject. We'll need to thread it through.

### 10. `FsWindow<E>`

A view-over-buffer implementation. 17 methods. Most are trivial iteration.

```java
public final class FsWindow<E> implements Window<E> {
  private final Object[] buffer;   // the underlying storage
  private final int      start;    // offset into buffer
  private final int      length;   // number of valid elements

  // Terminal ops: iterate buffer[start..start+length-1]
  public int size() { return length; }
  public boolean isEmpty() { return length == 0; }
  public E first() { return length == 0 ? null : (E) buffer[start]; }
  public E last()  { return length == 0 ? null : (E) buffer[start + length - 1]; }
  public void forEach(Consumer<? super E> action) { ... }
  public boolean all(Predicate<? super E> p) { ... }
  // ...

  // Restriction ops: produce new Window views over the same buffer
  public Window<E> prefix(int n) { return new FsWindow<>(buffer, start, Math.min(n, length)); }
  public Window<E> suffix(int n) { ... }
  public Window<E> skip(int n) { ... }
  public Window<E> slice(int from, int to) { ... }
  public Window<E> reverse() { ... }   // returns view with negative stride? or copy?
  public Window<E> trim(...) { ... }   // signature TBD
  // ...
}
```

**Reverse** can be implemented either as a strided view (stride = -1) or as an explicit copy. The strided-view approach preserves the "no copies" promise. Use a flag `reversed` and adjust iteration order accordingly.

**trim** — need to check the exact signature (Phase 1's grep didn't capture parameters). Will resolve when implementing.

### Implementation order

1. **FsWindow** — needed by FsFlow.window. Standalone, no dependencies. Easy ~150 LOC.
2. **FsFlow.window(int count)** — simple, depends on FsWindow.
3. **FsFlow.window(Duration, int)** — timestamp variant.
4. **FsFlow.scan** (both forms) — biggest design lift; depends on materialization machinery.
5. **FsFiber.every(Duration)** — additive, simple operator class.
6. **FsCircuit.pipe(Name, Receptor)** — additive, minor.
7. **FsFlow.flow(Function<Subject, Flow>)** — needs subject-aware materialization; design check during impl.
